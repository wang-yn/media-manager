# 元数据匹配弹窗设计

## 状态

- 日期：2026-07-09
- 状态：设计已批准，等待实现计划
- 范围：将媒体库详情里的元数据匹配从行内候选改为弹窗交互，并在弹窗内提供可编辑搜索词、候选选择、自动重命名预览和确认执行。

## 背景

当前用户点击“刮削”后，候选结果直接插入在媒体行下方。这个交互能工作，但空间有限，也不适合让用户修改匹配关键字后重新搜索。

本轮目标是让手工匹配更明确：

- 匹配结果出现在弹窗中。
- 弹窗提供搜索输入框，默认显示当前媒体标题。
- 用户可以修改关键字并重新匹配。
- 用户选择候选后，可以选择写入元数据并预览自动重命名后的文件名。
- 真实重命名仍需要用户看到目标路径后确认。

## 目标

- 点击“刮削”打开元数据匹配弹窗。
- 弹窗打开时用当前媒体标题作为关键字发起搜索。
- 弹窗顶部提供关键字输入框和“重新匹配”按钮。
- 搜索结果只在弹窗内展示。
- 用户点击候选后写入 `.nfo`。
- 弹窗提供选项：`写入元数据后自动重命名`。
- 勾选自动重命名时，写入 `.nfo` 后调用重命名预览，并在弹窗中显示自动重命名后的目标路径。
- 真正执行重命名前必须再次点击确认按钮。

## 非目标

- 不做批量匹配。
- 不做自动选择第一条候选。
- 不做分页、过滤、排序。
- 不下载海报或图片。
- 不引入 UI 组件库。
- 不引入 React Router。

## 交互流程

### 打开弹窗

用户在媒体库详情页点击某行的“刮削”。

前端状态：

- 当前媒体条目作为 `activeMetadataItem`。
- 搜索关键字初始为该媒体条目的 `title`。
- 弹窗打开后立即搜索。

### 修改关键字重新匹配

用户可以编辑输入框内容并点击“重新匹配”。

搜索规则：

- 关键字不能为空。
- 搜索中禁用“重新匹配”按钮。
- 搜索失败时在弹窗内显示后端结构化错误。
- 搜索成功时替换当前候选列表。

### 选择候选

候选列表展示：

- 标题。
- 年份。
- 简介。
- 媒体类型。

用户点击候选后：

- 调用 `metadata/apply` 写入 `.nfo`。
- 成功后刷新媒体列表。
- 如果未勾选自动重命名，则关闭弹窗。
- 如果已勾选自动重命名，则继续调用 `rename/preview`。

### 自动重命名预览

勾选 `写入元数据后自动重命名` 时，候选应用成功后弹窗保留打开状态，并展示重命名预览。

预览内容：

- `can_apply` 状态。
- 冲突列表，如果存在。
- 每个变更的目标路径。

如果没有冲突，显示“确认重命名”按钮。

### 确认执行重命名

用户点击“确认重命名”后：

- 调用 `rename/apply`。
- 成功后刷新媒体列表并关闭弹窗。
- 失败时保留弹窗并显示后端结构化错误。

## API 变化

保留现有 API，不新增端点。

允许 `POST /api/media/{id}/metadata/search` 接收可选 JSON body：

```json
{
  "query": "Pantheon"
}
```

兼容规则：

- 不传 body 时继续使用当前扫描标题。
- `query` 为空或全空白时回退当前扫描标题。
- 年份和媒体类型仍来自当前 `MediaItem`。

继续复用：

- `POST /api/media/{id}/metadata/apply`
- `POST /api/media/{id}/rename/preview`
- `POST /api/media/{id}/rename/apply`

## 状态模型

前端保留一个当前弹窗状态，不支持同时打开多个弹窗：

- 当前媒体条目。
- 搜索关键字。
- 候选列表。
- 自动重命名是否勾选。
- 重命名预览结果。
- 弹窗内错误。
- 当前忙碌动作。

关闭弹窗时清空这些状态。

## 错误处理

- TMDB 缺少密钥、请求失败、NFO 写入失败、重命名冲突都显示在弹窗内。
- 错误文案继续使用后端结构化错误的 `message/detail/path`。
- 全局错误条保留给页面级加载或设置页操作，不承载弹窗内错误。

## 文件安全

- 自动重命名只自动生成预览，不自动执行文件移动。
- 执行重命名必须由用户点击“确认重命名”。
- 后端继续在执行前重新计算预览并校验冲突。
