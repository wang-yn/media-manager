# 媒体库问题筛选与批量操作设计

## 状态

- 日期：2026-07-10
- 状态：设计已批准，等待实现计划
- 范围：在媒体库详情中筛选缺少元数据、缺少字幕或命名不规范的内容，并通过前端引导队列分别执行批量刮削、批量字幕和批量重命名。

## 背景

当前版本已经支持电影和剧集扫描、TMDB 手工刮削、ASSRT 字幕下载、自动重命名、目录大小、详细文件和剧集聚合展示。所有操作仍以单个媒体条目为入口，媒体较多时，用户难以快速找出待整理内容，也需要重复打开相同弹窗。

本轮把“发现问题”和“逐项修复”连接起来，但继续保持同步操作和人工确认：

- 用户先按问题类型筛选媒体。
- 用户选择电影或整部剧。
- 用户分别启动批量刮削、批量字幕或批量重命名。
- 需要选择候选时逐项确认，不自动采用第一条结果。

## 目标

- 支持筛选缺少元数据、缺少字幕和命名不规范的媒体。
- 支持选择当前筛选结果以及清空选择。
- 电影以单部为选择单位，剧集以整部剧为选择单位。
- 批量刮削、批量字幕和批量重命名使用三个独立入口。
- 元数据和字幕操作复用现有弹窗，当前项完成后自动进入下一项。
- 批量重命名先汇总预览，再由用户统一确认。
- 操作结束后显示成功、失败和跳过数量。
- 所有失败保留后端返回的详细错误信息。

## 非目标

- 不做后台任务或持久化任务队列。
- 不自动选择 TMDB 或 ASSRT 搜索结果的第一项。
- 不持久化筛选条件和已选择内容。
- 不做失败自动重试。
- 不保证多个媒体之间的事务回滚。
- 不新增数据库、状态管理库或 UI 组件库。
- 不实现一键执行元数据、字幕和重命名的完整修复流程。

## 整体方案

采用前端引导队列：

- 后端继续提供同步、无状态的媒体和文件操作 API。
- 前端保存当前筛选、选择、队列位置和本次结果。
- 批量元数据和字幕按顺序调用现有单项 API。
- 批量重命名复用现有单项执行和整剧执行 API，只补充整剧预览端点。
- 页面刷新、切换媒体库或取消批量时，未执行队列直接丢弃。

该方案不引入任务存储和轮询协议，也不改变当前手工确认的安全边界。

## 媒体问题状态

`GET /api/media` 为每个媒体条目补充以下字段：

- `has_metadata`：是否存在媒体级元数据。
- `rename_needed`：按当前重命名规则是否存在实际路径变化。

`has_metadata` 的判断规则：

- 电影检查视频所属电影目录内的 `movie.nfo`。
- 剧集检查剧集根目录内的 `tvshow.nfo`。

现有 `has_nfo` 字段保持原义，用于表示当前视频或单集对应的 NFO 是否存在，不替代 `has_metadata`。

`rename_needed` 复用重命名预览规则。只有标准化后的源路径和目标路径不同时才为 `true`；预览中仅包含同路径记录时为 `false`。状态计算不移动文件。

## 问题筛选

媒体库详情提供三个可多选筛选项：

- 缺少元数据：`has_metadata` 为 `false`。
- 缺少字幕：电影当前视频没有旁路字幕；剧集中至少一个单集没有旁路字幕。
- 命名不规范：电影或剧集中至少一个视频的 `rename_needed` 为 `true`。

筛选规则：

- 未选择问题类型时显示全部内容。
- 同时选择多个问题类型时，显示命中任意一个问题的内容，即 OR 关系。
- 修改筛选条件时清空当前选择，避免对不可见内容执行批量操作。
- “全选当前结果”只选择当前媒体库内通过筛选的可见电影或剧集。

## 选择粒度和批量工具栏

- 电影列表每行代表一部电影，复选框选择该电影。
- 剧集一级列表每行代表整部剧，复选框选择整部剧，不在季度或单集层级重复选择。
- 选中内容后显示批量工具栏：`批量刮削`、`批量字幕`、`批量重命名`、`清空选择`。
- 没有可处理目标的操作按钮保持禁用。
- 切换媒体库或离开媒体库详情时清空选择和未执行队列。

## 批量刮削

批量刮削只把缺少媒体级元数据的已选内容放入队列：

- 电影以当前电影条目进入队列。
- 剧集以整部剧的代表条目进入队列，每部剧只出现一次。
- 已存在媒体级元数据的已选内容记为跳过，不重复刮削。

队列复用现有元数据弹窗：

- 弹窗显示当前进度，例如 `2 / 8`。
- 搜索关键字、重新匹配、候选列表和自动重命名选项保持现有行为。
- 用户确认当前候选并完成写入后，自动打开下一项。
- 用户可以跳过当前项，或取消剩余队列。

## 批量字幕

批量字幕只处理没有旁路字幕的视频：

- 已选电影没有字幕时加入队列。
- 已选剧集展开为其中所有缺少字幕的单集，并按季、集顺序加入队列。
- 已有字幕的视频不重复加入队列。

队列复用现有字幕弹窗：

- 弹窗显示当前进度。
- 用户可以修改关键词、重新搜索并选择候选。
- 下载成功后自动进入下一项。
- ASSRT 配额或网络错误不会自动重试，当前弹窗保留并显示详细错误。

## 批量重命名

批量重命名先收集所有已选内容的预览：

- 电影调用现有 `POST /api/media/{id}/rename/preview`。
- 剧集调用新增的 `POST /api/media/{id}/rename/batch/preview`。
- 汇总弹窗按电影或剧集分组显示修改前后路径，并继续高亮差异部分。
- 已经符合规范的内容显示“已经是规范名称，无需修改”。
- 有冲突的内容显示冲突详情，不进入执行列表。

用户统一确认后，前端顺序执行：

- 电影调用 `POST /api/media/{id}/rename/apply`。
- 剧集调用现有 `POST /api/media/{id}/rename/batch`。
- 每个执行端点继续在修改文件前重新计算预览并检查冲突。
- 一个项目失败不回滚已成功项目，也不阻止后续项目执行。

## 弹窗和队列控制

元数据和字幕队列弹窗底部提供：

- `确定`：执行当前操作，成功后进入下一项。
- `跳过`：不修改当前内容，进入下一项。
- `取消批量`：停止所有剩余项目。

右上角 `X` 与“取消批量”行为一致。取消时已经成功的操作保留，当前项和剩余项不执行。

当前项失败时：

- 弹窗保持打开。
- 显示后端结构化错误的 `message`、`detail` 和 `path`。
- 用户可以重试、跳过或取消批量。

队列完成或取消后刷新媒体列表，并显示本次成功、失败和跳过数量。失败汇总保留媒体标题和具体错误。

## API 变化

### 媒体列表

`GET /api/media` 的每个条目增加：

```json
{
  "has_metadata": true,
  "rename_needed": false
}
```

两个字段均由后端根据当前文件状态实时计算，不写入配置或缓存。

### 整剧重命名预览

新增：

```text
POST /api/media/{id}/rename/batch/preview
```

该端点：

- 只接受剧集条目的 ID。
- 找出同一剧集下的所有扫描单集。
- 调用现有 `preview_batch_rename()`。
- 返回 `can_apply`、`conflicts` 和 `changes`。
- 不创建目录，不移动文件，不清理空目录。

批量元数据和字幕不增加后端批量端点。

## 错误和部分成功

- 单项搜索或写入失败时保留当前弹窗，不自动跳过。
- 用户主动跳过的项目计入跳过，不计入失败。
- 重命名预览冲突计入失败，并且不执行对应项目。
- 重命名执行期间出现文件系统错误时，保留后端原始错误详情。
- 批量操作允许部分成功；最终汇总明确列出每个失败项目。
- 页面级媒体加载失败继续使用全局错误区域，批量操作错误显示在对应弹窗内。

## 文件安全

- 元数据和字幕仍由现有写入函数限制目标位置。
- 重命名必须先展示预览并由用户确认。
- 执行端点必须重新计算预览，不能直接信任前端提交的目标路径。
- 批量预览端点不得产生任何文件系统修改。
- 取消操作不尝试回滚已完成的文件写入或重命名。

## 实现边界

预计修改：

- `backend/src/media_manager/media.py`：增加 `has_metadata` 字段和媒体级元数据路径判断。
- `backend/src/media_manager/server.py`：从重命名预览派生 `rename_needed`，并增加整剧重命名预览端点。
- `backend/tests/test_server.py` 和相关现有测试：状态和预览行为。
- `frontend/src/App.tsx`：筛选、选择、三个批量流程和结果汇总。
- `frontend/src/style.css`：筛选控件、批量工具栏和队列进度样式。

不新增后端模块、前端状态管理库或测试框架。优先复用现有扫描、弹窗、请求、错误解析和重命名函数。

## 测试与验证

后端自动测试覆盖：

- 电影存在或缺少 `movie.nfo` 时的 `has_metadata`。
- 剧集存在或缺少 `tvshow.nfo` 时的 `has_metadata`。
- 当前路径符合或不符合规范时的 `rename_needed`。
- 整剧重命名预览返回所有单集变更。
- 整剧重命名预览发现目标冲突。
- 调用整剧预览后文件和目录保持不变。

前端构建验证：

```bash
cd frontend && npm run build
```

手工验收覆盖：

- 单选和组合问题筛选。
- 全选当前结果和清空选择。
- 电影和整部剧的选择粒度。
- 元数据队列的确认、跳过、取消和失败重试。
- 剧集字幕队列按季集顺序展开。
- 批量重命名汇总预览、规范名称提示、冲突和部分成功。
- 队列结束后的列表刷新和结果汇总。

全量验证：

```bash
PYTHONPATH=backend/src .venv/bin/python -m unittest discover backend/tests
cd frontend && npm run build
git diff --check
```

## 设计自检

- 问题筛选、选择粒度和多筛选 OR 语义已明确。
- 三种批量操作保持独立，没有引入一键完整修复。
- 元数据和字幕候选继续由用户逐项确认。
- 剧集元数据按整部剧处理，字幕按缺失单集展开。
- 重命名在执行前提供汇总预览并重新校验。
- 取消、失败、跳过和部分成功语义已明确。
- 未引入后台任务、数据库、缓存或新依赖。
